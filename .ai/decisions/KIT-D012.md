Decided: the workflow tracks a **graph of work**, not flat stores. Builds on the id scheme
[[KIT-D011]] (`<KEY>-<TYPE><NUM>`).

## Scopes (the KEY)
- Per-project: `HOD` (hustle-or-die), `KIT` (claude-kit).
- **`LAB`** — a repo-less scope for cross-cutting ideas/research/planning not tied to any
  project. Lives in `claude-kit-data/` (no junction, no repo); surfaces in rundowns with no git
  state. Same atomic stores + scheme + generated index as any project.

## Item types (the TYPE letter)
- `R` **Request** — a planning-level ask. (hustle-or-die's existing `R###` ARE requests — this
  retires the wrong `R→T` collapse from KIT-D011's first pass: re-key `HOD-T### → HOD-R###`.)
- `E` **Epic** — groups multiple tickets under a request. *Optional* layer — a request may
  spawn tickets directly.
- `T` **Ticket** — a concrete task, broken out of a request (or epic).
- **Bug** — a Ticket with `type: bug`; not its own type. Its provenance links are the point.
- `D` decision, `N` note, `Q` question — supporting.

## Hierarchy
`Request → (Epic →) Ticket`. Each item stores ONE upward `parent:`; a request spawns many.
Downward views (a request's tickets, an epic's children) are GENERATED by the indexer — single
source of truth, no drift (same principle as the generated INDEX/REGRESSIONS).

## Relationship graph (frontmatter fields; upward-stored, downward-generated)
- `parent:` — request/epic this belongs to.
- `regression_of:` + `causing_commit:` — a bug regresses prior delivered work *(exist)*.
- `introduced_by:` — initial-development provenance; links **both** the ticket and the commit
  whose code the bug lives in *(new)*.
- `fixed_commit:` / `links:` — resolution + generic see-also *(exist)*.

## Documents as work products (artifact provenance)
Docs (research/design/strategy/CODEMAP) stay **path-addressed files** in `docs/` (NOT pulled
into the `.ai` store / id space). They carry provenance frontmatter:
- `produced_by:` — the work item that created the doc (one origin).
- `informs:` — the requests/tickets it seeds (many).
The indexer generates the reverse view (a ticket shows the docs it produced; a request shows the
docs that informed it). Full walkable loop: `Request → Epic → Ticket → commit → Document →
informs → next Request`. **Open-core:** `produced_by`/`informs` are private metadata — the
backport procedure STRIPS them when a doc is gifted to the public repo.

## General annotation
- `aka:` — list of an item's prior labels (e.g. `aka: [R045]`), carried on every re-keyed item
  so old references stay discoverable. Not migration-specific.

Rejected: docs as first-class id'd items (parallel id space, overhead); Bug as its own type
(it's work — a ticket); mandatory Epic layer (only when a request is genuinely large).
